# Contributing to Tider

Tider is open-source software, and contributions are welcome — bug reports, diagnosis rules, documentation, tests, and code. This guide walks you through the workflow end to end.

## Before you start

- Check the [issue tracker](https://github.com/tiderops/tider/issues). Picking an existing issue avoids duplicated or out-of-scope effort.
- For anything non-trivial (new feature, dependency, or architectural change), open an issue first to discuss the approach. The roadmap's "explicitly deferred" list is a delivery control — PRs that add SaaS infrastructure, write actions against clusters, or new frameworks will not be merged for Release 1.
- Vulnerabilities go through [SECURITY.md](./SECURITY.md), not public issues.

## Development setup

Requirements: Go 1.24, Node 26.04.0, Wails CLI 2.10 ([install guide](https://wails.io/docs/gettingstarted/installation)).

```bash
git clone https://github.com/<your-fork>/tider.git
cd tider
wails dev
```

You need a reachable Kubernetes cluster in your kubeconfig; a local [kind](https://kind.sigs.k8s.io/) or [k3d](https://k3d.io/) cluster works fine.

## Workflow

1. Fork the repository and create a branch from `develop`.
2. Make your changes in small, focused commits.
3. Use conventional commit prefixes: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`.
4. Run the quality gates (below) locally before pushing.
5. Open a pull request describing **what** changed and **why**. Link the issue it addresses.
6. Respond to review feedback — observations are about the code, not about you.
7. Once approved, your PR gets merged. Congratulations, and thank you!

## Quality gates

CI runs these; please run them locally first.

```bash
make check
npm run type-check
npm run lint
npm run test:unit
npm run test:e2e
```

Run a single frontend test with `npx vitest run path/to/spec.ts`.

## Architecture ground rules

The [README](./README.md#architecture-at-a-glance) has the full picture; these are the rules reviewers will hold you to:

- **Respect the layering.** `internal/binding` (interface) → `internal/usecase` (application) → `internal/k8s` (infrastructure). `internal/usecase` owns its port interfaces in `usecase/ports.go` and must never import `internal/k8s`.
- **Errors** returned from bound methods go through `apperr.Normalize` so the frontend receives stable `"KIND: message"` categories.
- **Context first.** `ctx` is the first parameter through the usecase and k8s layers; bound methods derive a per-call context with a 30s timeout.
- **Never edit `frontend/wailsjs/`.** It is regenerated by `wails dev`/`wails build` when Go signatures change; stage the regenerated files with `git add -u`.
- **Kubernetes access stays read-only.** Do not add patch/delete/exec/apply paths.
- **Frontend is feature-based.** New pages live in `frontend/src/features/<domain>/`; adding a resource kind to the UI is one `ResourceDescriptor` file, a `registry.ts` entry, and a route.
- **Diagnosis findings need evidence.** New troubleshooting rules must reference the exact evidence fields they use and handle the missing-evidence case rather than asserting a root cause.

## Style

- Go: `gofmt` (via `make fmt`) and `golangci-lint` clean.
- Frontend: Prettier + ESLint (`npm run format`, `npm run lint`).
- Match the conventions of the surrounding code; comments explain *why*, not *what*.
